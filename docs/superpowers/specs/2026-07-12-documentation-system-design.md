# DreamChord 文档体系设计

更新日期：2026-07-12

## 目标

让第一次接触 DreamChord 的人不用先理解代码，也能找到适合自己的入口：体验者可以启动项目，创作者可以完成一个故事，HR 或评审可以判断项目价值，开发者可以理解架构，维护者可以按固定流程发布新版本。

## 设计原则

- README 负责导览，不承担全部细节。
- 同一事实只保留一个权威出处，其他文档使用链接和短摘要。
- 先讲真实可用的工作流，再讲技术实现。
- 明确区分当前能力、已知边界和后续计划，不把路线图写成已实现功能。
- 命令统一从仓库根目录执行，版本号以根 `package.json` 为准。
- 面向普通读者的文档使用中文、短句和具体示例；必要的类型名、路径和命令保留英文原文。

## 读者入口

| 读者 | 第一入口 | 读完后应获得什么 |
|---|---|---|
| 第一次使用的小白 | `docs/GETTING_STARTED.md` | 能启动、登录、创建场景并完成预览 |
| 故事创作者 | `docs/CREATOR_WORKFLOW.md` | 理解故事、素材和 Agent 如何协同 |
| HR、老师、评审 | `docs/PROJECT_OVERVIEW.md` | 能快速判断产品价值、技术难点和项目完成度 |
| 开发者 | `docs/ARCHITECTURE.md` | 理解模块边界、数据流、关键约束和扩展点 |
| 贡献者 | `CONTRIBUTING.md` | 能搭建环境、修改、测试并提交合格变更 |
| 发布维护者 | `docs/RELEASE_GUIDE.md` | 能升级版本、维护变更记录并完成发布验证 |
| 安全研究者 | `SECURITY.md` | 知道支持范围、敏感数据边界和报告方式 |

## 信息架构

`README.md` 只保留产品定位、核心工作流、Agent 差异、截图、快速启动和文档导航。详细教程进入 `docs/`。项目治理文件留在仓库根目录，便于 GitHub 自动识别。

`docs/README.md` 是文档目录，也是文档维护规则。现有 Agent、长篇写作、场景设计、角色、素材和演示剧情文档继续保留，并从目录中按主题链接。

## 版本维护

- 当前版本：根 `package.json`、前后端包和共享包保持一致。
- 已发布变化：`CHANGELOG.md`。
- 未来方向：`docs/ROADMAP.md`。
- 发布步骤：`docs/RELEASE_GUIDE.md`。
- 架构事实：`docs/ARCHITECTURE.md`；更细的 AI 交接仍放在 `docs/AI_HANDOFF.md`。

## 验收标准

- 所有新增 Markdown 链接都指向存在的文件。
- README 在不阅读代码的情况下能说明产品解决什么问题。
- 新手文档包含从安装到完成一次预览的连续步骤。
- 项目概览不使用无法验证的商业数据或夸大表述。
- 路线图与当前能力分开。
- 发布指南包含版本号、数据库、测试、截图、变更记录和 Git 标签检查。
- 文档文本采用 UTF-8，命令与实际 `package.json` 脚本一致。
