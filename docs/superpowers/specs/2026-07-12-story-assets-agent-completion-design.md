# DreamChord 故事、素材与 Agent 完整改造设计

日期：2026-07-12  
状态：已批准（用户授权按推荐方案自主设计并直接实施）

## 目标

本轮改造让 DreamChord 的核心创作闭环真正一致：用户上传图片后能够理解图片是否适合做立绘、CG 或背景，得到安全的本地处理建议，并让 Agent 调用同一套检查与处理工具；用户在故事工作台看到、编辑和预览的官方演示必须是同一份剧情；官方演示本身需要有明确目标、分支后果和完整结尾。

本轮不伪装未实现的能力。默认安装不承诺对复杂摄影背景做语义分割，也不让语言模型直接访问用户电脑的文件选择器。用户负责选择并上传文件，服务器负责确定性分析和处理，Agent 负责理解结果、调用工具并提交可审阅产物。

## 方案选择

采用“共享故事图 + 本地确定性素材分析 + Agent 工具化”的方案。

- 不采用只修 seed 和页面文案的最小修补，因为编辑器与播放器仍会保留两份互相漂移的官方剧情。
- 不默认绑定云端视觉分割或生图服务，因为这会引入费用、密钥、隐私和离线启动问题。
- 为未来的视觉模型保留服务边界，但当前默认功能必须在一键启动后的本地环境中完整可用。

## 素材智能处理

### 输入分析

`imageInspector` 在基础尺寸和格式之外返回可解释的图像分析：

- 透明像素占比与是否已有有效透明通道；
- 四边像素的平均色、亮度与颜色方差；
- 是否疑似白色或浅色纯色底；
- 主体边界范围、主体占画面比例和宽高比；
- 推荐用途：立绘、CG 或背景；
- 推荐处理参数、置信度和中文警告。

分析是确定性的，结果随素材 API 返回，并存入处理变体的 metadata。Agent 的 `inspect_asset` 调用同一服务，不再只返回数据库中的文件名、尺寸和类型。

### 安全去底

白底去除改为“从图片边缘开始的连通区域抠除”。只有与外边界连通、且颜色满足阈值的浅色像素会被视为背景；角色服装、眼白、高光、纸张等被主体包围的白色区域不会被全局删除。羽化只作用于已识别背景边界附近。

输入建议分为三级：

1. 透明 PNG：最佳，默认关闭去白底；
2. 纯白或均匀浅色背景：可本地处理，默认开启边缘去底并显示置信度；
3. 复杂背景：不自动声称可以抠图，提示改用透明 PNG、纯色底原图或以后配置外部视觉服务。

所有处理结果仍是“候选变体”，用户或 Agent 都不能静默覆盖原图。接受后才绑定为角色表情或加入 CG/背景素材库；拒绝时保留原图并清理未引用的候选文件。

### 处理工作室

处理弹层打开时根据分析预选用途和参数，顶部显示“适合做什么、为什么、可能有什么问题”。用户可切换用途和手动调节阈值、羽化、裁边；任何参数变化都需要重新生成预览后才能接受。立绘接受时继续要求角色名和表情名。

## Agent 融合

Agent 保持现有的多对话、删除对话、分层记忆、运行时间线、补丁审阅与撤销能力，本轮补齐素材认知闭环：

1. `list_project_assets` 找到项目素材；
2. `inspect_asset` 读取真实像素分析与推荐；
3. `prepare_character_asset`、`prepare_cg_asset` 或 `prepare_background_asset` 生成候选变体；
4. 对话中展示工具调用、分析摘要和候选产物；
5. 用户接受后才写入正式素材和角色绑定。

Agent 系统提示必须明确素材能力边界：透明 PNG 最佳，均匀浅色底可自动处理，复杂背景不应盲目开启去白底。Agent 不直接上传本地文件，而是在用户上传后继续处理，这保证浏览器权限与审计逻辑清晰。

## 官方故事与工作台

### 单一可信数据源

在 `@dreamchord/story-domain` 中新增官方演示图定义，包含稳定 ID、场景标题、场景组、节点数据、边和正确的 `choice-N` 分支句柄。数据库 seed 从该定义创建或刷新系统演示项目；播放器和编辑器都通过公开项目 API 读取数据库图，再使用现有转换器播放。

删除 `dreamchord-first-thread` 在播放器中的硬编码剧情特判。公开且已发布的项目继续支持匿名试玩，私有草稿只允许作者预览。

seed 只刷新固定 ID 且固定系统作者拥有的官方演示项目，不修改任何用户项目。刷新时替换演示章节的节点、边和官方角色数据，使已有安装重新运行 seed 后也能获得新版本，而不是被 `update: {}` 永久锁在旧内容。

### 剧情重写

官方演示名为《第一根弦：消失的终页》。核心目标是：轻小说部将在日落时被审查，雪必须找回被抹去的终页，完成社团最后一份作品。

剧情按九个有标题的场景推进：

1. 稿纸归零：建立日落期限和“终页被删除”的异常；
2. 教室残响：雪看见现实中的故事节点；
3. 樱花路上的影子：被废弃的旧主角质问雪为何停止写作；
4. 咖啡馆的真相：米娅揭示雪因害怕失败而主动删掉结局；
5. 星弦空间：找到没有名字、因此无法做出选择的角色；
6. 命名：雪将其命名为空，并承认角色也应拥有意志；
7. 分支回响：早先选择改变雪面对旧稿、朋友与失败的态度；
8. 写下终页：最终选择决定结局主题，而非跳转产品宣传；
9. 日落之后：社团作品完成，当前冲突闭环，并自然引导用户进入工作台继续创作。

首个三选一分支必须在图上显示三个明确去向，每条分支至少有独立对白、状态写入和后续回调。最终分支可以汇合，但汇合后的文本必须反映之前选择。每个场景节点写入 `sceneTitle` 和 `sceneGroupId`，保证场景树不再出现一串“角色对话”。

## 错误处理与兼容性

- 图像无法解码、像素超限或文件丢失时返回可读错误，且不创建数据库候选记录。
- 分析置信度低时保持手动参数，不自动接受结果。
- 旧素材没有分析 metadata 时按需即时分析，不要求数据库迁移。
- 官方演示刷新失败时事务整体回滚，避免留下半章数据。
- 播放器加载失败显示明确错误和返回入口，不进入空白或假完成状态。

## 测试与验收

自动化测试至少覆盖：

- 边缘白底被移除，但主体内部白色块保留；
- 透明 PNG、纯色底立绘、宽屏背景和复杂底图得到合理且稳定的分析建议；
- Agent `inspect_asset` 返回分析，处理工具沿用推荐参数且仍生成候选变体；
- 官方图不存在悬空边、空对白、错误分支句柄或无标题场景；
- seed 刷新官方演示但不触碰普通用户项目；
- 播放器对官方演示与普通项目走同一数据加载路径。

浏览器验收覆盖桌面和移动视口：上传透明 PNG、白底人物和复杂背景图片；检查推荐、预览、接受/拒绝；从工作台查看九个场景和三条分支；完整试玩至少两条路线，确认分支回响、结局、返回工作台和无重叠布局。验收后删除测试上传及未接受变体。

## 完成标准

- 用户能清楚知道上传什么图片效果最好，并在不破坏白色服饰的前提下生成透明立绘；
- Agent 能检查真实素材、解释建议、调用处理工具，并把产物留给用户确认；
- 官方演示在编辑器、预览和公开试玩中完全一致；
- 场景导航标题、选择分支和剧情逻辑清晰；
- 新用户可通过一键启动完成安装、seed、登录、创作、素材处理和试玩；
- 全量测试、构建和浏览器验收通过后再提交并推送 GitHub。
