# DreamChord 创作 Agent 与产品完善设计

## 1. 背景

DreamChord 已具备项目、章节、镜头卡、节点图、播放器、素材库和基础 AI 能力。现有 AI 由润色、续写、选项生成、分支回应和节点图生成等独立入口组成，能够完成单次文本任务，但存在以下问题：

- 每次调用只获得临时拼接的项目摘要，缺少稳定的世界观、人物弧光、文风和禁忌记忆。
- 模型直接返回文本或宽松解析的节点 JSON，没有计划、工具边界、变更预览和可靠校验。
- 独立 AI 写作页与编辑器 AI 面板职责重叠，用户需要在多个入口间切换。
- 未配置模型时返回看似智能的固定文本，容易让用户误判功能状态。
- AI 生成的节点可以直接合入编辑器，但缺少事务快照、结构差异和一键撤销。
- 前端没有测试框架，图结构变更、上下文选择和 AI 状态流缺少回归保护。
- `WorkbenchPanel.tsx` 体积过大，继续把 Agent 状态和界面放入其中会扩大维护风险。

本阶段以“从创作意图到可审查、可播放的剧情变更”为唯一主线，不扩展协作、商业化或素材生成等外围能力。

## 2. 目标与非目标

### 2.1 目标

1. 将零散 AI 功能统一为可理解项目、制定计划、调用确定性工具和提出结构化变更的单 Agent。
2. 让所有作品修改都经过差异预览、确定性校验和用户确认，并支持完整撤销。
3. 为项目增加持久的“故事圣经”，让 AI 在角色、世界观和文风上保持一致。
4. 增加规则驱动的剧情体检，使未配置模型时仍有真实、可解释的辅助能力。
5. 合并重复 AI 入口，降低认知负担，同时保留常用任务的一键快捷方式。
6. 为 Agent 核心和图变更补齐自动化测试，并控制新增模块边界。

### 2.2 非目标

- 不实现多 Agent 编排。
- 不允许 Agent 自动发布作品、删除项目或绕过确认写入。
- 不实现联网资料搜索、RAG 向量库、图片生成或语音生成。
- 不重写播放器、素材库、认证或现有项目保存协议。
- 不在本阶段实现多人协作、评论增强、模板市场或付费系统。

## 3. 成功标准

端到端主路径为：用户在编辑器中输入“检查第二章并补一个合理分支”，Agent 能够：

1. 明确显示作用项目和章节。
2. 读取故事圣经、第二章结构、相邻章节摘要和相关角色。
3. 展示简短计划及实际工具执行进度。
4. 输出剧情诊断和一份结构化节点变更集。
5. 在应用前显示新增、修改、删除和连线变化。
6. 通过结构、引用、分支和内容限制校验。
7. 仅在用户点击“应用”后一次性写入。
8. 应用后能够一键撤销到完全相同的前置状态。

附加验收要求：AI 不可用时，剧情体检仍可运行；Agent 失败时不得留下半写入状态；所有关键纯逻辑模块有测试。

## 4. 产品结构

### 4.1 统一入口

编辑器右侧保留一个“创作 Agent”入口，替代当前独立的 AI 助手模式列表。面板包含：

- 任务输入框：接收自然语言目标。
- 作用域选择：当前镜头、当前场景、当前章节、全项目。
- 快捷任务：润色当前、续写场景、生成选项、修复分支、剧情体检。
- 运行时间线：计划、工具调用、诊断、草案、校验和完成状态。
- 变更预览：按新增、修改、删除分类，并支持跳转到镜头卡。
- 确认区：应用、放弃、重新生成；应用成功后显示撤销。

原 `AIWriterPage` 改造成 Agent 的全屏任务视图，共用相同会话和运行状态，不再维护另一套提示词和草稿逻辑。导航中的名称改为“创作 Agent”。

### 4.2 故事圣经

项目设置新增“故事圣经”页签，字段为：

- 世界观摘要
- 叙事主题
- 文风说明
- 时间与地点规则
- 禁止事项
- 角色目标、秘密、关系和语言习惯

角色基础资料继续以现有 `Character` 为权威来源；故事圣经只补充叙事约束，不复制姓名、颜色和立绘字段。

### 4.3 剧情体检

体检分为确定性规则和 AI 建议两层。确定性规则无需模型：

- 入口不可达的场景或节点
- 没有目标的选项
- 无出口且非结局的场景
- 引用不存在角色或素材的节点
- 选择后立即汇合、分支没有实质差异
- 连续过长对话、长期没有背景或角色状态更新
- 空文本、重复节点 ID、悬空边和非法节点字段

AI 层只分析角色一致性、情绪转折、节奏和伏笔，不得覆盖确定性规则结论。每条问题包含严重度、解释、位置和可选修复动作。

### 4.4 删除与合并

- 删除 `AIAssistantPanel` 中的固定文本“本地 AI”回退；改成明确的规则辅助状态。
- 删除前后端重复的自然语言列表解析路径；Agent 只使用结构化响应。
- 合并 `AIWriterPage` 与编辑器助手的提示词、配置读取和请求状态。
- 旧 `/ai/polish`、`/ai/continue`、`/ai/choices` 暂时保留一版兼容期，由快捷任务内部转向 Agent，后续再删除。
- 不删除播放器、节点编辑器和镜头卡编辑器中的专业模式。

## 5. Agent 架构

### 5.1 原则

采用“单 Agent + 确定性工具 + 人工确认”的架构。模型负责理解目标、选择工具、组织计划和生成创作草案；程序负责权限、上下文获取、图操作、ID、校验、持久化和撤销。

Agent 不直接获得数据库或文件系统访问，不接受任意工具名称，不执行模型产生的代码。

### 5.2 状态机

运行状态固定为：

`queued -> planning -> gathering_context -> drafting -> validating -> awaiting_approval -> applying -> completed`

任意执行状态可进入 `failed` 或 `cancelled`。只有 `awaiting_approval` 可以进入 `applying`。`completed` 的写入任务必须关联一个可撤销快照。

服务器限制每次运行最多 8 个工具步骤、最多 2 次结构修复、最多 60 个新增节点。超过限制返回可继续的草案，不自动扩大权限或循环。

### 5.3 工具目录

工具使用固定注册表和 Zod 输入输出模式：

- `read_project_brief`：读取项目摘要、故事圣经和角色约束。
- `read_chapter_outline`：返回章节场景列表、入口、出口和摘要。
- `read_scene`：读取指定场景的镜头卡和图结构。
- `search_story`：按角色、文本或节点类型检索，返回有限结果。
- `analyze_story_graph`：运行确定性诊断，只读。
- `create_story_patch`：把创作草案转换为白名单图操作。
- `validate_story_patch`：验证补丁，不写入。

工具结果只返回任务需要的字段，避免把整个项目无条件塞进上下文。`create_story_patch` 生成的 ID 由服务器替换为 `crypto.randomUUID()`，模型提供的 ID 只能作为临时引用。

### 5.4 结构化变更集

Agent 不返回完整章节覆盖，而返回操作列表：

```ts
type StoryPatchOperation =
  | { kind: 'addNode'; tempId: string; node: ProposedNode; anchor?: PatchAnchor }
  | { kind: 'updateNode'; nodeId: string; changes: AllowedNodeChanges }
  | { kind: 'removeNode'; nodeId: string }
  | { kind: 'addEdge'; sourceRef: string; targetRef: string; sourceHandle?: string }
  | { kind: 'removeEdge'; edgeId: string }
```

`AllowedNodeChanges` 按节点类型定义字段白名单，禁止任意 `Record<string, unknown>` 写入。位置可以由 Agent 提议，但最终通过现有布局规则归一化。

### 5.5 上下文策略

上下文按作用域组装：

1. 始终包含任务、故事圣经摘要和相关角色约束。
2. 当前镜头任务包含前后各 4 张镜头卡。
3. 场景任务包含完整场景和相邻场景摘要。
4. 章节任务包含章节大纲，只在工具请求后读取完整场景。
5. 全项目任务先读取章节摘要，禁止默认发送所有节点。

每段上下文带稳定来源标识，界面可以显示“Agent 阅读了什么”。

## 6. 数据模型

新增以下 Prisma 模型：

- `StoryBible`：项目一对一，保存结构化 JSON 和版本号。
- `AgentConversation`：属于用户和项目，保存标题、作用域及时间。
- `AgentMessage`：保存用户、助手和工具摘要；不保存 API Key 或完整供应商请求。
- `AgentRun`：保存状态、任务、计划、错误、模型信息和用量摘要。
- `StoryPatch`：保存结构化补丁、校验结果、应用状态和基准章节版本。
- `ChapterSnapshot`：保存应用前的节点边快照，用于一次完整撤销。

`Chapter` 增加整数 `version`。Agent 草案记录基准版本；应用时必须匹配，否则提示项目已变化并重新校验，防止覆盖用户并发编辑。

SQLite 中不持久化明文 API Key。第一阶段继续由浏览器保存用户配置并随 Agent 请求传给服务端内存使用，同时确保请求体、错误和运行记录不落盘。现有未使用的 `AIProviderConfig` 模型从 schema 移除；如果未来需要跨设备同步，另行设计加密密钥服务。

## 7. API 设计

新增接口：

- `GET /api/projects/:projectId/story-bible`
- `PUT /api/projects/:projectId/story-bible`
- `GET /api/projects/:projectId/agent/conversations`
- `POST /api/projects/:projectId/agent/conversations`
- `POST /api/projects/:projectId/agent/runs`
- `GET /api/agent/runs/:runId`
- `POST /api/agent/runs/:runId/cancel`
- `POST /api/agent/runs/:runId/apply`
- `POST /api/agent/runs/:runId/reject`
- `POST /api/agent/runs/:runId/retry`
- `POST /api/agent/runs/:runId/undo`
- `GET /api/projects/:projectId/health`

第一阶段使用普通 HTTP 和前端短轮询获取运行进度，不引入 WebSocket。运行步骤会持久化，刷新页面后仍能恢复。每个变更接口验证登录用户是项目作者。

`POST /runs` 返回 `202 Accepted` 和运行 ID。服务端在受控执行器中运行 Agent。首版可以使用进程内队列，但队列接口必须独立，便于以后替换持久任务系统。

## 8. 应用、事务与撤销

补丁应用流程：

1. 开启 Prisma 事务。
2. 校验作者权限、运行状态和章节版本。
3. 再次运行补丁验证。
4. 写入 `ChapterSnapshot`。
5. 按依赖顺序应用节点和边操作。
6. 递增章节版本，标记补丁已应用。
7. 提交事务。

任何一步失败则整笔回滚。撤销要求章节仍处于该补丁应用后的版本；若用户已经继续编辑，系统不做破坏性覆盖，而是要求先查看冲突。

## 9. 校验与错误处理

### 9.1 输入校验

- 所有新 API 使用 Zod，拒绝未知字段。
- 自定义 Base URL 只允许 `http` 或 `https`，生产环境默认拒绝本机和私网地址，降低 SSRF 风险。
- 限制输入长度、消息数量、工具结果大小和补丁操作数。

### 9.2 图校验

补丁必须满足：

- 节点和边引用存在且 ID 唯一。
- 节点类型和字段符合现有运行时协议。
- 选择节点的 `sourceHandle` 与选项索引一致。
- 删除节点时同步处理关联边。
- 新增内容从指定锚点可达。
- 非结局场景不能因补丁产生意外断路。
- 生成图可被 `converter.ts` 转换并被 `runtime.ts` 遍历。

### 9.3 模型错误

- 结构解析失败：把模式错误摘要反馈给模型，最多修复两次。
- 供应商超时或限流：保留运行和已收集上下文，允许重试。
- 内容为空或越权：标记失败，不生成可应用补丁。
- 部分工具失败：停止后续写操作，展示具体失败步骤。

错误信息不得包含 API Key、Authorization 头或完整供应商响应。

建议新增：

- `src/agent/AgentPanel.tsx`：容器和路由级编排。
- `src/agent/AgentComposer.tsx`：任务输入、作用域和快捷任务。
- `src/agent/AgentTimeline.tsx`：运行步骤。
- `src/agent/PatchPreview.tsx`：结构差异和定位。
- `src/agent/AgentApprovalBar.tsx`：应用、放弃、重试和撤销。

`WorkbenchPanel.tsx` 只保留工作区布局与面板切换。现有 AI 模式到快捷任务的映射放入独立适配器，迁移完成后删除旧状态和固定回退文案。

## 11. 测试策略

引入 Vitest。测试按风险排序：

### 11.1 纯逻辑单元测试

- 上下文选择不会越过作用域或无限膨胀。
- 每种补丁操作的归一化和字段白名单。
- 图校验覆盖悬空边、错误选项句柄、不可达场景和删除关联边。
- 确定性剧情体检规则。
- Agent 状态机禁止非法跳转。

### 11.2 服务端集成测试

- 未授权用户不能读取或应用运行。
- 章节版本冲突不会写入。
- 应用失败完整回滚。
- 应用后撤销恢复相同节点和边。
- 模型返回非法结构时触发有限修复并最终安全失败。

### 11.3 前端组件测试

- 不同运行状态只显示允许的命令。
- 差异预览正确分类和定位。
- 未配置模型时明确显示规则辅助状态。
- 刷新后可恢复待确认运行。

### 11.4 端到端验证

使用可控假模型完成主验收路径，不依赖真实 API。另用手动冒烟测试验证至少一个 OpenAI 兼容供应商。

## 12. 实施阶段

### 阶段 A：可信底座

测试框架、故事圣经、章节版本、补丁类型、诊断规则、图校验、事务应用和撤销。

### 阶段 B：Agent 执行器

工具注册表、上下文策略、状态机、运行存储、模型结构化协议、有限重试和 API。

### 阶段 C：统一体验

Agent 面板、全屏视图、时间线、差异预览、快捷任务和旧 AI 入口迁移。

### 阶段 D：收尾

拆分 `WorkbenchPanel`、删除固定回退和重复提示词、兼容性验证、文档与端到端验收。

每一阶段都必须保持现有编辑、保存和播放流程可用。

## 13. 关键取舍

- 选择单 Agent 而非多 Agent，因为当前任务的核心是可靠修改一个共享故事图，多角色协作会增加延迟、成本和不可预测性。
- 选择结构化补丁而非整章覆盖，因为补丁可审查、可校验、可撤销，也更容易避免覆盖用户工作。
- 选择服务端确定性工具而非让模型直接操作数据库，因为权限和图一致性不能依赖提示词。
- 选择短轮询而非 WebSocket，因为首版运行量有限，持久运行记录已经解决刷新恢复问题。
- 选择浏览器暂存密钥而非继续使用数据库明文字段，因为当前没有可靠的服务端加密密钥管理。
- 选择规则体检作为无模型降级能力，因为固定生成文案不真实，也无法为作品提供可靠价值。

## 14. 完成定义

本功能只有在以下条件全部满足时才算完成：

- 主验收路径可用并有自动化覆盖。
- Agent 的任何写入都需要确认、事务和版本校验。
- 应用和撤销经过集成测试。
- 无模型时不展示伪 AI 结果。
- 旧快捷功能在统一入口中仍可访问。
- 前后端构建通过，新增测试通过。
- 文档说明数据隐私、模型配置、限制和失败恢复行为。
