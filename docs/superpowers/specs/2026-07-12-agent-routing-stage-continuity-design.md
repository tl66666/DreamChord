# Agent 路由与舞台连续性设计

## 目标

DreamChord 的特色不是单独放一个聊天页，而是让 Agent 真正理解项目、帮助创作，并且所有写入行为都有清楚边界。故事编辑器同时要用视觉小说作者熟悉的“舞台状态”工作，而不是要求每条对话重复设置背景和人物。

## Agent 执行边界

每条消息先进入确定性路由，默认只读：

1. **本地即时回答**：问候、时间日期、能力说明、项目概览、角色/素材盘点和确定性剧情体检。无需 API Key，也不调用外部模型。
2. **只读对话**：一般知识、创作讨论、主题分析、没有绑定章节的续写构思。模型可以自然返回中文，也可以调用读取类工具；不能生成剧情补丁或素材衍生图。
3. **受控行动**：绑定章节后的明确续写/润色/分支修改，以及明确的素材处理请求。模型可调用白名单工具，产物进入预览和审批，不能直接写入故事。

路由的安全原则是“默认只读，明确动作才写候选”。模型即使在只读对话中误返回 `patch`，系统也会丢弃补丁并保留文字回答。执行异常必须写入一条用户可见的 Agent 消息，不能只在任务记录中失败。

## 无模型状态

未配置 API Key 时，本地即时能力继续可用。开放式知识问答和生成任务返回明确的能力边界与设置入口，不伪造模型结果，也不把对话标记成失败。

## 舞台连续性

视觉小说的背景和角色是沿剧情路径持续存在的舞台状态：

- 背景节点替换当前背景，后续镜头默认继承。
- 角色 `show` 表示登场或更新表情/位置，`hide` 表示退场，没有事件则继续保持。
- 新建下一镜头时，编辑器计算前一节点之后的有效舞台，把背景和仍在场角色作为“继承”状态展示。
- 新人物登场不会清掉旧人物；旧人物只有明确选择退场才消失。
- 场景卡展示最终舞台结果，而不是只统计当前场景组里新建的角色节点。
- 分支内沿本分支继承；分支汇合出现不一致时不武断选择，显示舞台冲突并提示作者在汇合点明确重置。

## 产品入口

- 首页首屏说明 Agent 能处理的真实任务，并为每个故事提供“问 Agent”入口。
- 独立 Agent 工作区继续承担多对话、记忆和工具审计。
- 编辑器保留上下文 Agent，用当前镜头/场景/章节直接发起任务。
- README、Agent 指南、展示页统一解释本地即时、只读对话、受控行动和舞台继承。

## 验收

- `现在几点` 在配置 GLM 时也不调用 GLM，直接得到当前时间。
- 一般问题接受自然文本首包，不经历三次格式修复。
- 项目对话中的错误补丁不会造成失败或写入故事。
- 模型网络错误会在对话中显示可理解回复。
- 新镜头继承上一镜头背景与角色；新增人物、退场和表情更新符合顺序状态。
- 桌面和移动端真实浏览器验收无重叠、无死操作，展示页引用本轮真实截图。
